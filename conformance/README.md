# JTF conformance vectors

Language-agnostic golden test vectors that **define** JTF spec compliance. They
are the contract every JTF implementation must satisfy.

## Layout

```
manifest.json          index of all cases
cases/NNN-name.json    input value (canonical JSON)
cases/NNN-name.jtf     expected JTF output (the contract)
run.py                 the conformance runner (Python + JS)
runner_js.mjs          JS side, invoked by run.py
_gen_inputs.py         regenerates cases/ + manifest.json from the Python lib
```

## The contract

For every case, a conformant implementation MUST satisfy **both**:

1. `encode(input.json)` equals `expected.jtf` byte-for-byte, using the
   **canonical heuristic cost function** (`max(1, floor(len/4))`, see SPEC §9.1).
2. `decode(expected.jtf)` equals `input.json` (lossless round-trip).

The Python library is the source of truth: the `.jtf` files are generated by it,
and the JS library is aligned to match.

## Coverage

The 45 cases cover: scalars (null/bool/int/float/string/empty), numbers and
floats incl. numeric-looking strings, null/empty containers, flat and nested
objects, deep nesting, uniform tabular arrays (flat and nested/dotted), keys
needing CSV headers, primitive/mixed/heterogeneous/array-of-array shapes, Unicode
(Cyrillic + emoji), strings needing escaping (`= : tab newline , " \ # $ @ |`),
and the value dictionary (exact `$N` entries incl. `$1`-vs-`$10`, URL prefix
aliases, and the not-profitable case).

> Integral floats (`45.0`) and float exponents (`1.5e10`) are intentionally
> excluded: a single-numeric-type host (JavaScript) collapses them to integers at
> JSON-parse time, so they are not cross-language representable. See SPEC §2.1.

## Run

```sh
./run-conformance.sh                 # from the repo root
# or
python3 conformance/run.py
```

## Regenerate (after an intentional, spec-aligned format change)

```sh
python3 conformance/_gen_inputs.py   # rewrites cases/ + manifest.json
./run-conformance.sh                 # confirm JS still matches
```
